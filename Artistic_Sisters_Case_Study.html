01 — Overview

What Is Artistic Sisters?

Artistic Sisters is an online marketplace built around the handmade art business. It lets customers browse a gallery of artworks, place orders, and also request custom commissions directly from artists. On the other side, artists manage their portfolio and review commission requests. Delivery agents handle logistics — they see their assigned shipments and update the delivery status in real time.

The platform was built to go beyond a basic CRUD app. It uses a proper microservices architecture where each part of the system — orders, payments, logistics, notifications — runs as a completely independent service. This makes the system more resilient: if the notification service goes down, order placement still works.

02 — Problem

What Problem Does It Solve?

The fine arts market is a difficult space to digitise. Unlike regular e-commerce where you pick a product and buy it, art can be one-of-a-kind, hand-crafted to order, or negotiated between buyer and artist. Most existing platforms are built for mass-market goods and don't handle any of this well.

Specific Gaps Addressed

  • Two different order types in one flow: Ready-made artwork that ships immediately needs a completely different workflow than a custom commission that requires artist approval and a production window. Both had to coexist cleanly in the same system.
  • Keeping data consistent across services: When an order is placed, the payment service needs to know, the logistics service needs to prepare, and the customer needs a confirmation email — all without these services calling each other directly.
  • Role separation: Artists should only see their own portfolio. Delivery agents should only see their assigned shipments. Customers should only see their own orders. This needs to be enforced server-side, not just in the UI.
  • Reliable notifications: An email that never arrives for a confirmed order is a bad user experience. The system needed to guarantee that events like "order confirmed" or "agent assigned" always trigger their notifications, even if the broker hiccups temporarily.
03 — Architecture

System Architecture

All client traffic goes through a single entry point — the Ocelot API Gateway on port 5000 — which routes requests to the right microservice. Every service has its own SQL Server database; no two services share a database. Cross-service communication happens exclusively through RabbitMQ events.

Microservice Responsibilities

:5001
Identity Service

Handles all registration and login flows for customers and delivery agents. Issues 24-hour JWT tokens signed with HMAC-SHA256. Passwords are always stored as BCrypt hashes — never in plain text.

:5002
Artwork Service

Manages the artwork catalog. Artists upload pieces with details like medium, price, dimensions, and available quantity. Customers can subscribe to "Notify Me" alerts for coming-soon items.

:5003
Order Service

The most complex service. Manages the full order lifecycle through 7 states. Handles both standard ready-made orders and custom commission requests. Uses Redis for reservation state and an Outbox pattern for reliable event delivery.

:5004
Logistics Service

Assigns delivery agents and vehicles to shipments when an order is ready for dispatch. Tracks delivery milestones and runs a Dispatch Saga for agent assignment with SLA monitoring.

:5005
Payment Service

Integrates with the Razorpay SDK to create payment orders and verify payment signatures. Also supports Cash on Delivery. Listens to order events to initiate and confirm payment flows.

:5006
Notification Service

A standalone email dispatcher. It doesn't expose any business endpoints — it just listens to RabbitMQ events and sends the right email to the right person. Customers, artists, and agents are all notified at relevant milestones.

04 — Backend Design

Clean Architecture & CQRS

Every backend service follows Clean Architecture. The domain layer — entities, enums, business rules — has zero dependencies on any framework. Infrastructure concerns like EF Core and MassTransit live in their own layer and depend inward on domain abstractions.

All operations go through MediatR. A controller never directly calls a repository. It dispatches a command or query, and MediatR routes it to the right handler. This keeps controllers thin and testable, and makes business logic easy to find.

How a Request Flows Through the System

  1. Gateway — Ocelot matches the route and forwards to the correct service.
  2. JWT Middleware — Token is validated, role claims are extracted into the request context.
  3. Controller — Binds the request body, checks [Authorize(Roles="...")], dispatches via IMediator.Send().
  4. FluentValidation — Validates the command before the handler runs. Bad input fails here with a structured 400 response.
  5. MediatR Handler — Runs business logic, mutates domain entities, prepares events.
  6. EF Core + Outbox — Domain changes and the outbox message are written in one SQL transaction.
  7. MassTransit Relay — Picks up the outbox message and publishes it to RabbitMQ for consumers.

Example: Registration Handler

IdentityService / Application / Commands / Register / RegisterHandler.cs
public async Task<RegisterResult> Handle(RegisterCommand request, CancellationToken ct)
{
    // Reject duplicate emails before doing anything
    var exists = await _db.Customers.AnyAsync(c => c.Email == request.Email, ct);
    if (exists)
        return new RegisterResult { Success = false, Message = "Email already registered" };

    // Password is always hashed — plain text never touches the database
    var customer = new Customer {
        Id           = Guid.NewGuid(),
        PasswordHash = BCrypt.Net.BCrypt.HashPassword(request.Password),
        IsActive     = true,
        RegisteredAt = DateTime.UtcNow
        // ... other fields
    };
    _db.Customers.Add(customer);
    await _db.SaveChangesAsync(ct);

    // Publish event so NotificationService sends the welcome email
    await _publisher.Publish(new CustomerRegisteredEvent {
        CustomerId = customer.Id, Name = customer.Name, Email = customer.Email
    });

    return new RegisterResult { Success = true, Token = GenerateToken(customer) };
}
05 — Event-Driven Design

Integration Events & Messaging

Services never call each other over HTTP. Everything is asynchronous through RabbitMQ via MassTransit. When something meaningful happens — an order is placed, an agent is assigned, a delivery is completed — the responsible service publishes an event. Any service that cares about that event subscribes to it independently.

The Outbox Pattern — Why It Matters

There's a well-known problem in distributed systems: what if you successfully save to the database but then fail to send the RabbitMQ message? The order exists but no one gets notified. The Outbox pattern solves this.

In the Order Service, when a handler runs, it writes the domain change and the serialised event into an OutboxMessages table — all in the same SQL transaction. A background relay process polls this table and publishes the messages to RabbitMQ, marking them as processed once done. If the relay crashes mid-flight, it simply retries — the message is still in the outbox.

Result: At-least-once delivery is guaranteed. No order confirmation, payment, or logistics event can go missing due to a transient broker failure.

Idempotency guard: The handler checks whether the order is already in ReadyForDelivery before publishing ReadyForDispatchEvent. If someone re-submits the same status update, no second dispatch event fires — preventing duplicate agent assignments.
07 — Custom Commissions

Commission Workflow

Custom commissions are the differentiator of this platform. Unlike a standard purchase, a commission requires a back-and-forth between the customer and artist before any payment happens.

  1. Customer submits request via POST /api/orders/commission — providing medium, size, instructions, and optionally a reference photo URL. The order is created with status Pending.
  2. Artist is notified via the CustomCommissionEvent that triggers an email from the Notification Service with all the commission details.
  3. Artist reviews on their dashboard and either accepts (sets a quoted price, moves to Confirmed) or declines (Cancelled). In both cases, an OrderStatusChangedEvent emails the customer.
  4. Customer accepts and pays — once confirmed with a price, the customer proceeds through the normal checkout: Razorpay online payment or Cash on Delivery.
  5. Standard fulfilment — the order then follows the same In Production → Ready → Dispatch → Delivered pipeline as any other order.
08 — Frontend

Angular 21 Frontend

The frontend is a standalone Angular 21 SPA. It uses reactive forms, RxJS streams, and a JWT interceptor that automatically attaches Authorization: Bearer <token> to every API request. The token, role, and userId are stored in localStorage under the keys as_token, as_role, and as_userId.

Guards check the role stored in localStorage. If the role doesn't match, the user is redirected to /auth/login. Server-side, the [Authorize(Roles="...")] attribute provides a second enforcement layer — so even if someone bypasses the frontend guard, they can't access protected API endpoints.

10 — Testing

Unit Tests

The project includes 14 unit tests across two test projects — IdentityService.Tests and OrderService.Tests. All tests use EF Core InMemory databases so they need no real SQL Server, Moq for mocking MassTransit's publish endpoint, and xUnit as the runner. Every test is fully isolated and deterministic.

12 — Summary

Key Takeaways

Artistic Sisters is not a prototype or a CRUD demo. It is a working, end-to-end platform with real payment processing, a proper microservices architecture, a tested business layer, and a functioning frontend across three distinct user roles.

The main engineering decisions — CQRS with MediatR, event-driven communication over RabbitMQ, the Outbox pattern for reliability, database-per-service isolation, and BCrypt + JWT for security — were all made deliberately to build something that would hold up in a production environment, not just in development.

  • Services are independently deployable — one failing doesn't take down the rest.
  • Events are never silently lost — the Outbox pattern guarantees delivery.
  • Business rules are enforced server-side — the frontend guards are a UX layer, not the security layer.
  • Every critical handler is covered by a focused, isolated unit test.
  • Custom commissions and standard orders share the same core pipeline, keeping the codebase unified.
